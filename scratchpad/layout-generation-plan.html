Generating a whole map
from pieces and interfaces

The layout generator separates a map's topology (the plan) from its geometry (the realization). The plan is areal by construction — rectangular pieces joined by edge-interval interfaces, never a thinned skeleton — and generation grows outward, rule-based, from the deliberately boring rectilinear seeds. This is that document, laid out in build order, with each stage marked by where it actually stands.

The core claim

Topology before geometry

A plan is a small, explicit, persisted artifact: globals (symmetry mode, team count, board bbox, a coarse 5-block grid cell), one authored per-team unit, and the neutral middle. It holds the meaning that a raw layout.json + intent.json pair only implies — roles, isolation, elevation transitions — and it compiles into both:

  • Piece — a rectangle on the coarse grid, tagged with a role (spawn, hub, lane, wool-room, mid, frontline), a plateau height, and pinned objective placements.
  • Interface — a shared edge interval between two pieces. Not a point: it has a position and a width. land merges the pieces; gap leaves a void a bridge must span. A piece with no land interface is isolated — one rule that expresses every isolation motif.
The master variable Interface width is the whole design. The lane — the seeds' 10-block / 2-cell corridor — is the reference; the rungs w2 / w4 / w6 (≈ 1 / 2 / 3 lanes) set connectivity, classify the joint (a touch ≤ 1 lane continues a lane; ≥ 3 lanes is a hub), and gate which fills are legal at that touch. Player-flow control is just the distribution of interface widths along the attack paths.
The box-based approach — visualized

A wool box is an envelope of typed rectangles

The terrain immediately around a wool room is a wool box: a bounding envelope subdivided into an ordered template of role-typed rectangles. A family (I, L, Z, scythe, U, H, clamp, donut) is the same fixed set of boxes, only resized. Below, two wool boxes — an I approach and a Donut approach. In both, the wool room box is filled (the objective is placed); the terrain boxes stay empty envelopes — drawn but not yet realized — and void is genuinely open space a build zone may later span.

Terrain boxes enclose a void — the strongest structural signal (a full loop, multi-access). The room box is filled on a tail off the ring; the ring stays empty envelopes; the centre is an enclosed hole. Same six boxes at any scale.

wool room — filled (placed objective) terrain box — empty envelope void — open / bridgeable enclosed hole (donut)

Why the split is load-bearing: the classifier reads a shape width-independently (turn count off the outline, branch off two legs sharing an edge, hole off an enclosed void), and the emitter builds each family as the same stable set of boxes tagged with a slot role. So the composition rules — entry widening, wool docking, which boxes may split into build zones — become properties of a role, defined once per family, instead of ad-hoc geometry. Both halves are already built and verified (see the board below).

Generation, in order · §3

What the generator does, step by step

The runtime pipeline. The composer's natural order is the author's gesture run forward: compose the map as one connected mass, assign the team sides, then fragment each regime. Everything runs in a relative frame from the spawn outward, then embeds into absolute coordinates once symmetry resolves the join.

1
Globals shipped
Symmetry, teams, board bbox, grid cell — sampled by Envelope from the player count (G8 land-budget coupling, G3 board bands).
2
Grow the team unit partial
Place spawn → attach hub → grow k lanes each dead-ending in a wool-room → attach frontlines. TeamUnitGrower + real spawn/wool-room rooms ship; the lane-archetype grammar and hub-shape variety remain (G37/G41).
3
Fragmentation — carve mid, cut team sides partial
Carve the mid into its interface form (clean / parallel / hash), cut secondary isolation behind bridges. MidCarver + IsolationCut ship the clean form; parallel/hash bands and the frontline↔build-zone interlock remain (G38/G39/G40).
4
Neutral middle partial
Place mid pieces and build zones between the frontlines. Clean band + CT7-snapped mid stones ship; multiple/parallel bands are unshipped (G38).
5
Validate the plan shipped
Zero-geometry invariants — every wool reachable, no wool path through a spawn, ≥1 gap per inter-team path, widths ≥ corridor minimum. PlanValidator gates every compose; full-suite goldens are the open slice (G32-D).
6
Heights not started
Per-role plateau defaults (raised spawn, stepped wool approach, low frontline), transitions per interface. Rooms are flat at the base surface today — this is the G32-C elevation pass.
7
Fan by symmetry & emit partial
ComposeGeometry fans the authored unit into teams islands. The full SketchLayout+MapIntent emit through the existing export chain, plus markers/walls, is the G32-C/D finish.
8
Roughen pass not started
Separate, last: turns the rectilinear plan into an organic read. See phase 4 below.

P4

Anchor jitter, edge subdivision + displacement, pull-to-polygon (the "twist"), width profile, 45° chamfer, piece shear/rotate — with per-operator invariants (corridor minimum, no self-intersection, interfaces stay covered). Turns the boring rectilinear plan into an organic read; runs before the author ever sees output.
P5
Elevation rules not started
Plateaus, cut & raise (refine one piece into two joined by a land interface with a height delta), role defaults, ramps/cliffs if in scope. Attached to roles and interfaces, not geometry. The plan carries height fields; the composer-side rule pass does not exist yet (overlaps G32-C).
P6
Mutation in the UI not started
Plan operators (add a wool branch, isolate a piece, widen an interface, re-side the spawn) + re-roll buttons per layer. The staged loop: edit the plan, re-roll roughen and elevation independently.

Scope & the division of labour

What the generator owns, what the author owns

"Making a boring layout interesting" is a generator stage, not user work — the roughen and elevation passes run before the author sees an output. The author's share is three smaller things: rule authorship (a mostly one-time 20–40-rule checklist), curation (reject outputs and say why — each articulated rejection becomes a checkable invariant), and post-detach polish (set pieces, themed builds — always manual).

  • Tier 1 (high confidence) — parameterized seed families end-to-end: a "new map" button producing a valid, walkable map that reads competent but plain. The first ~60–70% of layout work.
  • Tier 2 (viable via curation) — re-roll workflows, plan-level mutation ops, rules converging from feedback rounds. Ceiling: maps tweaked for an hour instead of built for days.
  • Tier 3 (not promised) — signature interestingness. Rule systems recombine encoded motifs; they don't coin new ones. The motif library grows only when the author adds to it.
Evaluation instrument · ground truth A month of server logging (pgmlogger → parquet) yields per-map traffic graphs: player positions every 2s, aggregated on a 3-block grid. Two readings feed generation — the closure photographed (hot void cells are the build regions emerging from behaviour alone) and flow priors to score composer candidates. Validated on ingwaz: the plan model, a real map, and real player behaviour agree to the block.

Why not the corpus, WFC, or skeletons. The 350-map corpus is reference imagery, not an oracle (extraction flattens away what made maps read). WFC solves local adjacency; CTW quality is global and relational. A medial-axis skeleton can't carry side-contact intervals and biases toward thin maps. The rule set is expert-authored, areal by construction, and grown from the boring seeds.
